# ClawBus Protocol (v0)

This document is the canonical specification for ClawBus messages and adapter behavior. Everything else in the repo is an implementation detail that must conform to this file.

Status: **draft, v0**. Expect breaking changes until v0.1.

---

## 1. Terminology

- **Agent** — any named process that sends or receives ClawBus messages. Usually a Claude Agent SDK session, but it can be any program (human, shell script, webhook handler).
- **Bus** — the in-process facade (`ClawBus` class) that agents talk to. Reads and writes to an adapter.
- **Adapter** — the transport + storage layer. Examples: FileAdapter (append-only JSONL on disk), SQLiteAdapter (single-file DB), DiscordAdapter (plugin).
- **Store** — the append-only record of all messages. Required by every adapter.

---

## 2. Message

A ClawBus message is a JSON object with this shape:

```typescript
type ClawBusMessage = {
  id: string;                 // ULID. Opaque. Generated by the bus on send.
  from: AgentId;              // string, [a-z0-9_-]+ recommended
  to: AgentId | "broadcast";
  kind: MessageKind;
  payload: unknown;           // kind-specific; see §4
  parent?: string;            // id of the message this is a response to
  createdAt: string;          // RFC3339 / ISO8601 UTC
  meta?: Record<string, unknown>; // open extension slot
};
```

### Invariants

1. `id` is assigned by the bus at `send()` time. Callers must not supply it.
2. `createdAt` is assigned by the bus. Callers must not supply it.
3. Messages are **immutable** once stored. To "edit" a message, send a new one with `parent` set.
4. `parent` must reference an existing message id in the same store, or be absent.
5. Broadcasts (`to: "broadcast"`) are delivered to every agent that has an active subscription.

---

## 3. Agent IDs

- Case-sensitive ASCII strings.
- Recommended pattern: `[a-z][a-z0-9_-]{0,63}`.
- Reserved ids:
  - `"broadcast"` — only valid as `to`, never as `from`.
  - `"system"` — reserved for bus-generated housekeeping messages. User agents must not send as `system`.

---

## 4. Message kinds

ClawBus v0 defines exactly five kinds. Keep the count small on purpose.

### 4.1 `task`

An agent asks another agent (or broadcasts) to do something.

```typescript
payload: {
  goal: string;                 // one-line human-readable goal
  context?: unknown;            // free-form structured context
  deadline?: string;            // ISO8601, optional
  budget?: { tokens?: number; steps?: number };
};
```

### 4.2 `result`

A response to a `task`. `parent` **must** be set to the task's id.

```typescript
payload: {
  status: "ok" | "failed" | "partial";
  summary: string;              // one-line human-readable outcome
  artifacts?: Array<{ kind: string; ref: string }>; // e.g. file paths, URLs
  error?: { message: string; code?: string };
};
```

### 4.3 `approval-request`

An agent asks for human (or supervisory agent) sign-off before taking a risky action.

```typescript
payload: {
  action: string;               // one-line description of what will happen
  diff?: string;                // unified diff, optional
  severity: "low" | "medium" | "high";
  rationale: string;            // why this is necessary
  defaultDeny?: boolean;        // if true, timeout = reject; else timeout = accept
  timeoutMs?: number;           // optional; 0 = wait forever
};
```

### 4.4 `approval-decision`

The response to an `approval-request`. `parent` must be set.

```typescript
payload: {
  decision: "approve" | "reject";
  reviewer: string;             // who decided (agent id or human identifier)
  note?: string;
};
```

### 4.5 `log`

Informational breadcrumb. Never expects a response.

```typescript
payload: {
  level: "debug" | "info" | "warn" | "error";
  text: string;
  data?: unknown;
};
```

## 5. Adapter contract

Every adapter MUST implement:

```typescript
interface Adapter {
  append(msg: ClawBusMessage): Promise<void>;
  subscribe(
    agentId: AgentId,
    handler: (msg: ClawBusMessage) => void | Promise<void>,
  ): Unsubscribe;
  query(filter: Partial<Pick<ClawBusMessage,
    "from" | "to" | "kind" | "parent">>): Promise<ClawBusMessage[]>;
  close(): Promise<void>;
}
type Unsubscribe = () => void;
```

### Delivery semantics

- `append` is **at-least-once durable** before it resolves.
- Handlers are called for messages where `to === agentId` or `to === "broadcast"`.
- Handler errors must be caught by the adapter and not crash the bus. They should be logged as `system`→`log` messages with `level: "error"`.
- Order: within a single `from`, messages are delivered in send order. Across `from`s, order is unspecified.

### Query semantics

- `query` returns messages in `createdAt` ascending order.
- All filters are AND-combined. Missing filters match anything.

---

## 6. Bus semantics

The `ClawBus` class wraps an adapter and adds ergonomic helpers:

- `send({ from?, to, kind, payload, parent?, meta? })` — generates `id` + `createdAt`, appends, returns the stored message.
- `subscribe(agentId, handler)` — pass-through to adapter.
- `waitFor({ parent?, kind?, from?, timeoutMs? })` — resolves with the first matching future message. Subscription-based; also scans past messages so late subscribers don't miss responses that arrived before they called `waitFor`.

The bus is **process-local**. Cross-process coordination is the adapter's job.

---

## 7. Security & rights

- Adapters are responsible for enforcing any access control needed for their transport. The protocol itself is open.
- Payloads may contain sensitive data (code, secrets). Adapters that persist to shared storage (e.g. DiscordAdapter) must document their redaction policy.
- Human approval gates exist so agents cannot unilaterally execute destructive actions. Implementations should surface `approval-request` prominently in every UI layer.

---

## 8. Versioning

This spec is v0. Every message **may** carry `meta.protocolVersion` (string). Absent means v0. Adapters must pass `meta` through unmodified.

A v1 bump is expected to add: batching, signed messages, and a distinguished `heartbeat` kind.
